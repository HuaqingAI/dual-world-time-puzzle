# Story 1.1: 启动到首个花园房间

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a 玩家,
I want 启动游戏后进入一个清晰的首个花园房间,
so that 我能立即理解自己在哪里、控制谁、以及房间里存在一个可探索目标。

## Acceptance Criteria

1. **Given** 玩家启动垂直切片构建  
   **When** 主场景加载完成  
   **Then** 游戏显示首个花园房间 shell，包含玩家起始点、房间边界、可读地面层和至少一个远处目标占位  
   **And** 不显示调试 UI 或复杂菜单遮挡场景

2. **Given** 首房间加载完成  
   **When** 玩家停留在入口附近  
   **Then** 摄像机 / 视口构图能同时显示玩家、主要通路方向和目标占位  
   **And** 在 16:9、1080p 基准下不需要滚动 UI 或额外说明才能理解房间方向

3. **Given** 开发者查看项目结构  
   **When** 检查 Godot 项目文件  
   **Then** 项目使用 Godot 4.6.2 + GDScript 的空项目基础  
   **And** 初始目录遵循架构约定，至少包含 `assets/`、`data/`、`scenes/`、`scripts/`、`docs/`

4. **Given** 首房间 shell 已加载  
   **When** 后续 stories 增加玩家移动、世界切换、种子和 UI  
   **Then** 当前场景结构能容纳 `Player`、`Objects`、`UI` 和后续 puzzle object 节点  
   **And** Story 1.1 不要求任何未来 story 才能成功启动和显示房间

## Dev Notes

### Story Scope

Story 1.1 是项目的第一条实现故事，目标是“启动后能看到一个可读的首房间 shell”。它不是移动、世界切换、种子收集、音频、暂停菜单或 UI 设计系统故事。

必须完成的玩家可见结果：运行项目后直接进入首个花园房间 shell；玩家起点、边界、地面层、主通路方向和远处目标占位同时成立。

明确不要做：

- 不实现完整玩家移动逻辑；Story 1.3 负责。
- 不实现 InputMap 全量动作；Story 1.4 负责。
- 不实现 `WorldStateManager`、双世界切换或藤蔓桥状态；Epic 2 负责。
- 不实现种子收集、跟随、出口花闭环或种植仪式；Epic 3 和 Epic 4 负责。
- 不提前实现 `EventBus`、`Log`、`AudioManager`、Theme Resource、UI 组件契约；Story 1.2 负责。
- 不创建复杂菜单、默认调试覆盖层、长教程文本、背包、经济、敌人、死亡、倒计时或追逐压力。

### Current Repository State

- 当前仓库没有 Godot 项目文件、场景或 GDScript 实现；实现范围主要是新增项目骨架和首房间 shell。
- 已存在 `_bmad-output/`、`_bmad/`、`.agents/`、`.claude/` 等规划和工具资料；不要移动或重写这些文件。
- 目前没有上一条故事文件；无需继承前一故事的实现模式。
- 最近 git 提交只涉及 BMad 安装和规划资料整理，没有可复用 gameplay 代码。开发者应以架构文档为主，不从工具目录复制运行时代码。

### Technical Requirements

- 引擎：Godot 4.6.2 stable。
- 语言：GDScript；不要使用 C#，因为项目架构以 Web demo 支持和 GDScript 为基线。
- 初始化方式：空 Godot 项目；命令参考 `godot --editor --path .`。
- 主场景：`res://scenes/main/main.tscn`。
- 首房间场景：`res://scenes/rooms/room_001.tscn`。
- 目标平台：PC / Steam 与 Web demo；首个 shell 应按 16:9、1080p 可读性设计。
- 渲染：Godot 2D renderer；优先避免昂贵 shader 和高噪声全屏特效。

### Architecture Compliance

项目结构必须向后兼容架构定义的系统位置：

- `scenes/main/`：主场景入口。
- `scenes/rooms/`：房间场景组合。
- `scenes/player/`：后续玩家场景。
- `scenes/puzzle_objects/`：后续藤蔓桥、花门、出口花等对象。
- `scenes/ui/`：后续互动提示、状态指示、菜单和反馈组件。
- `scenes/debug/`：后续调试覆盖层，默认不得在 Story 1.1 启动时显示。
- `scripts/player/`、`scripts/ui/`、`scripts/debug/`、`scripts/gameplay/`：保留脚本领域边界。

推荐 `room_001.tscn` 节点组织：

```text
Room001 (Node2D)
├── Ground
├── Bounds
├── PlayerSpawn
├── PlayerPlaceholder
├── Objects
│   └── DistantGoalPlaceholder
├── Camera2D
└── UI
```

如果开发者选择创建 `player_placeholder.tscn`，应放在 `scenes/player/`，只负责视觉占位和后续替换位置，不承载移动或交互逻辑。

### File Structure Requirements

新增文件建议范围：

- `project.godot`
- `scenes/main/main.tscn`
- `scenes/rooms/room_001.tscn`
- 可选：`scenes/player/player_placeholder.tscn`
- 可选：`docs/implementation-notes.md` 或 `docs/scene-structure.md`，仅在需要记录启动验证和结构说明时创建

不要在 Story 1.1 中新增运行时代码到 `scripts/autoloads/`，除非 Godot 项目生成流程强制需要；共享全局服务属于 Story 1.2。

### UX Guardrails

- 首屏不要像应用落地页或菜单页；应直接显示可玩的花园房间 shell。
- 入口构图必须表达“我在哪里、控制谁、目标大概在哪个方向”。
- 远处目标占位优先采用微光、轮廓、形状或轻微动画感，避免只靠颜色。
- UI 默认最小化；不显示地图、任务列表、长教程或调试信息。
- 画面密度服务可读性，装饰植物和占位块不得遮挡主通路、玩家占位或目标占位。

### Testing Requirements

最低验证：

- Godot 4.6.2 打开项目无报错。
- 运行项目直接加载 `scenes/main/main.tscn`，并显示 `room_001`。
- 启动画面包含玩家起点、房间边界、可读地面层、远处目标占位。
- 默认不显示 Debug Overlay、复杂菜单或技术错误。
- 1920x1080 下玩家、主要通路方向和目标占位同屏可见。
- 1280x720、1600x900 下构图仍可读，没有 UI 溢出或遮挡关键对象。

验证命令参考：

```bash
godot --path .
godot --headless --path . --quit-after 1
```

如果 Godot executable 不在 PATH，开发者必须在 Dev Agent Record 中说明，并至少完成文件结构和 `project.godot` 静态检查。

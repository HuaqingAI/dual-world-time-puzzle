# 微光花园 / Seedlight Garden - Game Design Document

**Author:** Sue  
**Game Type:** Puzzle / 解谜游戏  
**Target Platform(s):** PC / Steam；Web demo 用于试玩、传播与早期反馈；移动端作为后续验证方向

---

## Executive Summary

### Core Concept

《微光花园 / Seedlight Garden》是一款休闲俯视角双世界花园解谜游戏。玩家扮演迷路的小精灵，在“枯萎”和“盛放”两个花园状态之间切换，观察植物、道路、藤蔓和花门的变化，找到发光种子，并把种子带回中心花园，让沉睡的花园逐步恢复生命、颜色和声音。

核心体验不是惩罚错误或考验反应，而是让玩家在低压力的观察、尝试和发现中获得“我修复了这里”的回报。

### Target Audience

主要受众是 30-40 岁、偏休闲取向的成年玩家，尤其是喜欢 cozy、轻量解谜、短时完整体验、低惩罚探索和温柔视觉反馈的玩家。游戏应支持通勤碎片时间、晚上放松和睡前短玩，一次标准房间目标体验控制在 4-8 分钟。

次要受众是喜欢精致小体量独立解谜和 cozy indie 的玩家。他们重视规则优雅、画面可读、关卡节奏、音画反馈和情绪一致性，可能熟悉《Monument Valley》《Carto》《A Short Hike》《Unpacking》《Dorfromantik》《Botanicula》《Samorost》等作品。

### Unique Selling Points (USPs)

1. **双世界是可读的修复工具。** 枯萎和盛放不是硬核时间谜题，而是玩家用来理解路径、唤醒对象、修复空间的低压力工具。
2. **发光种子是情感对象。** 种子不是普通钥匙；它被收集后成为短暂陪伴玩家的小光点，并在种下时带来花园恢复反馈。
3. **一房间一个小魔法。** 每个标准房间围绕一个清晰变化或一个主要规则设计，入口尽量让玩家看到目标种子。
4. **表现反馈承担教学、确认和奖励。** 颜色、形状、动画、音频和微光轨迹共同说明状态变化，减少文字教学负担。

## Goals and Context

### Project Goals

1. 做出一个高完成度垂直切片，验证“切换世界改变路径”和“种下种子修复花园”的核心闭环。
2. 建立可扩展但不膨胀的解谜结构：每个房间一个主要规则，每个区域一个主机制族。
3. 用可读的视觉语言定义枯萎 / 盛放差异，让玩家不依赖颜色也能理解关键状态。
4. 优先打磨世界切换、藤蔓桥、花门、发光种子、出口花和种植仪式，再扩展关卡数量。
5. 为后续 `gds-game-architecture` 和 `gds-create-epics-and-stories` 提供明确的机制、关卡、反馈和史诗边界。

### Background and Rationale

项目来自 `dual-world-time-puzzle` 的方向探索。头脑风暴和 Game Brief 已收敛到“迷路花园精灵”概念：休闲俯视角、双世界状态切换、发光种子收集、中心花园修复、单房间短闭环。

关键判断是：项目成败不在谜题数量，而在两个体验是否成立。第一，玩家能否在短时间内理解“切换世界会改变路径或对象状态”。第二，玩家把种子带回并种下时，是否明确感到“我让这个地方变好了”。

## Core Gameplay

### Game Pillars

#### 1. 温柔修复优先

每个核心关卡都应让玩家看到花园从沉睡、枯萎或缺失转向恢复、盛放或重新发光。解谜完成后的变化不是附加奖励，而是游戏承诺的一部分。

设计约束：

- 不使用敌人、死亡、硬倒计时或惩罚性失败作为主线压力。
- 错误尝试可逆；玩家可以重置房间或继续尝试。
- 每颗主线种子都必须带来可见、可听或可交互的恢复反馈。

#### 2. 清晰可读的双世界变化

枯萎和盛放必须通过颜色、形状、轮廓、动画、声音和对象状态共同区分。关键变化要能在切换后的 1 秒内被玩家注意到，尤其是路径、藤蔓桥、花门、出口花、种子和提示对象。

设计约束：

- 重要状态不能只靠颜色区分，必须至少同时改变形状、轮廓、动作或发光状态。
- 每次切换最多强调 1-2 个与当前谜题直接相关的对象。
- 状态变化服务路线理解，不做纯装饰的高噪声变化。

#### 3. 一房间一个小魔法

标准房间围绕一个主要规则或一个可记住的变化设计。入口尽量让玩家看到发光种子或目标方向，谜题重点是“如何到达那里”，不是让玩家在复杂迷宫或长链路中迷失。

设计约束：

- 教学房间只引入 1 个新规则。
- 普通房间可以组合 1 个已学规则和 1 个当前规则。
- 可选挑战房间才能使用 3 个以上规则组合。

#### 4. 表现反馈服务玩法

动画、音频、粒子、颜色和 UI 反馈必须帮助玩家理解状态变化、交互结果和成长进度。美术表现不只负责好看，也负责教学、确认和奖励。

设计约束：

- 世界切换、藤蔓桥展开、花门苏醒、种子跟随、出口花开启和种植仪式是第一优先级表现对象。
- 音频必须区分枯萎 / 盛放状态，并确认关键交互。
- UI 只补足必要信息，不替代表现反馈。

### Core Gameplay Loop

1. 玩家进入一个紧凑花园房间。
2. 玩家在入口附近看见发光种子、出口花或目标方向。
3. 玩家切换枯萎 / 盛放世界，观察路径、藤蔓、花门和植物状态变化。
4. 玩家利用当前房间的主要规则打开路线。
5. 玩家收集发光种子，种子以微光轨迹跟随玩家。
6. 玩家带种子到出口花或中心花园入口。
7. 玩家种下种子，触发 2-4 秒的恢复反馈。
8. 中心花园新增一处永久变化，提示下一颗种子的方向或下一房间入口。

垂直切片目标时长：第一次完整闭环 3-5 分钟。标准主线房间目标时长：4-8 分钟。后期或可选房间目标时长：8-12 分钟。

### Win/Loss Conditions

**房间成功条件：** 收集当前房间的发光种子，并把它带到出口花或种植点完成种植反馈。

**整体进度条件：** 种下足够数量的主线种子后，中心花园恢复新的区域入口、视觉层次或章节路径。

**失败状态：** 主线体验没有死亡、战斗失败或硬倒计时失败。玩家可能进入“暂时卡住”状态，但可以通过观察提示、手动重置房间、离开房间或使用提示继续推进。

**重置规则：** 重置只恢复当前房间的谜题对象和玩家位置，不清除已完成的中心花园永久恢复。

## Game Mechanics

### Primary Mechanics

| Mechanic | Player Action | Rule | Feedback | Pillar Trace |
| --- | --- | --- | --- | --- |
| 俯视角移动 | 玩家控制小精灵穿过花园房间 | 移动应支持细微调整，玩家能在 8 秒内穿过一个标准小房间的主通路 | 脚步微光、草叶轻动、边界轻柔阻挡 | 一房间一个小魔法 |
| 世界切换 | 玩家按一个主按钮在枯萎 / 盛放之间切换 | 切换转场 0.45-0.75 秒；转场期间不接受重复切换；切换不造成伤害或失败 | 盛放涟漪、音乐层淡入淡出、关键对象高亮 | 清晰可读的双世界变化 |
| 观察变化 | 玩家比较两个世界的路径和对象状态 | 当前谜题关键对象在切换后 1 秒内出现视觉或音频提示 | 对象轮廓、发光、开合动画、短音效 | 表现反馈服务玩法 |
| 藤蔓桥 | 玩家在盛放状态通过藤蔓桥 | 枯萎状态不可通行；盛放状态展开并可通行 | 藤蔓 0.6-0.9 秒展开，叶片发光 | 清晰可读的双世界变化 |
| 花门 | 玩家靠近或携带种子唤醒花门 | 睡眠状态阻挡；唤醒后打开路线或出口 | 花瓣呼吸、门心亮起、开启音 | 温柔修复优先 |
| 发光种子 | 玩家触碰或互动收集种子 | 种子在主线路径上可见；收集后不占背包格，表现为跟随光点 | 收集闪光 0.3-0.5 秒，跟随微光轨迹 | 温柔修复优先 |
| 出口花 / 种植 | 玩家带种子到出口花或中心花园种植点 | 有种子时出口花开启；种植后完成房间 | 2-4 秒种植仪式、颜色恢复、音频层增强 | 温柔修复优先 |
| 房间重置 | 玩家从暂停菜单或重置点恢复房间初始状态 | 重置不惩罚、不清除已完成主进度 | 柔和淡出淡入，保留已种下种子记录 | 低压力可逆实验 |

| --- | --- | --- | --- |

## Puzzle Game Specific Design

### Core Puzzle Mechanics

《微光花园》的核心谜题是二态世界切换带来的路径和对象状态变化。玩家不是在抽象棋盘上求最优解，而是在一个小花园房间中观察“枯萎”和“盛放”如何改变可通行路线、对象状态和目标可达性。

#### Puzzle Elements

| Element | Withered / 枯萎 | Bloom / 盛放 | Player Meaning | Slice Priority |
| --- | --- | --- | --- | --- |
| 藤蔓桥 | 卷起、断裂、不可通行 | 展开、发光、可通行 | 切换世界改变路线 | 必须有 |
| 花门 | 闭合、沉睡、阻挡路线 | 可被唤醒或开启 | 需要找到正确状态或携带种子 | 必须有 |
| 发光种子 | 目标可见但可能不可达 | 目标更亮，提示路线 | 房间目标和情绪对象 | 必须有 |
| 出口花 | 关闭或低亮度 | 携带种子后开启 | 带种子离开房间 | 必须有 |
| 花瓣路标 | 稀疏、暗淡 | 更清晰、轻微飘动 | 低压力方向提示 | 应有 |
| 露水 / 水面 | 可能阻挡或隐藏路径 | 可能露出浅滩或反射提示 | 后续区域规则 | 后续 |
| 月光植物 | 静止或不可互动 | 对光线 / 状态响应 | 后续组合规则 | 后续 |

#### Constraint Systems

- 每个教学房间只测试一个主要对象规则。
- 主线房间的必要操作链控制在 3-6 个关键步骤。
- 前 3 个主线房间不要求玩家记忆屏幕外对象状态。
- 关键目标应在入口 3 秒内可见，或通过花瓣路标明确暗示。
- 玩家可以随时切换世界观察，不消耗资源。

### Puzzle Progression

| Stage | Goal | Rule Load | Target Solve Time | Example |
| --- | --- | --- | --- | --- |
| Tutorial | 学会移动、看见种子、种下种子 | 0-1 个规则 | 2-4 分钟 | 直线路径 + 种植仪式 |
| Core Concept | 学会切换世界改变路线 | 1 个规则 | 3-5 分钟 | 藤蔓桥在 Bloom 展开 |
| Reinforcement | 学会花门 / 出口花状态 | 1 个新规则 + 1 个旧规则 | 4-8 分钟 | 先切换过桥，再唤醒花门 |
| Combined | 组合多个对象状态 | 2 个主要规则 | 6-10 分钟 | 藤桥、花门、花瓣路标组合 |
| Optional Challenge | 提供更高理解密度 | 3 个以上规则 | 8-12 分钟 | 可选种子或明信片房间 |

难度来自阅读关系和规划顺序，不来自高操作精度、反应速度、惩罚、隐藏像素点或长链记忆。

### Level Structure

垂直切片包含：

1. 一个中心花园小块，显示尚未恢复和恢复后的对比。
2. 一个完整花园房间，包含小精灵移动、世界切换、藤蔓桥、花门、发光种子、出口花和种植反馈。
3. 一个简短开始状态和一个结束状态，支持外部试玩。

[ASSUMPTION A1: 完整版 v1 以 4 个区域、16-24 个主线房间、4 个区域恢复节点为规划上限；该数字用于设计和生产估算，不是当前垂直切片承诺。]

候选区域结构：

| Area | Theme | New Motif | Emotional Payoff |
| --- | --- | --- | --- |
| 根之角落 | 基础花园、藤蔓、土壤 | 藤蔓桥与花门 | 花园重新长出第一片路 |
| 露水池塘 | 水面、倒影、露珠 | 水位或露水路径 | 安静水声和微光倒影恢复 |
| 月亮温室 | 月光植物、玻璃、夜色 | 光线状态和开合节奏 | 夜间花朵点亮温室 |
| 风之花架 | 风、花瓣、轻移动 | 花瓣路标和风向提示 | 花架重新发声、花瓣回流 |

### Player Assistance

玩家辅助采用分层提示，不把答案直接写在屏幕上。

| Layer | Trigger | Behavior |
| --- | --- | --- |
| Passive Cue | 玩家进入房间 | 目标种子、出口花或关键对象保持轻微发光 / 动作 |
| Observation Cue | 玩家切换世界 | 与当前谜题相关的 1-2 个对象短暂高亮 |
| Idle Nudge | 玩家 45-60 秒无有效进展 | 花瓣、种子微光或小精灵视线指向关键区域 |
| Manual Hint | 玩家主动请求 | 提供一句温和提示，先提示观察方向，再提示操作顺序 |
| Reset | 玩家主动选择 | 当前房间恢复初始状态，不惩罚进度 |

[ASSUMPTION A3: 完整版可允许玩家在连续使用多层提示后跳过非教程主线谜题，但垂直切片先不实现跳过，以免掩盖核心规则可读性问题。]

### Replayability

重玩价值来自重访恢复后的空间、欣赏前后对比、补看环境变化和收集轻量纪念物，而不是主线计时、步数压榨或排行榜。

主线 v1 不追求：

- 完美步数挑战。
- 每日谜题。
- 程序生成关卡。
- 排行榜。
- 强制速通评价。

可选重玩方向：

- 回到已修复房间查看盛放后的细节变化。
- 收集无压力的花园明信片或种子图鉴片段。
- 在中心花园查看每颗种子带来的永久变化。

## Progression and Balance

### Player Progression

玩家进展由“种下发光种子”驱动。种子不是货币，不能消费，也不进入复杂背包；它们是恢复进度、章节解锁和情绪回报的核心标记。

进度层级：

1. **房间内进展：** 找到路线，收集当前种子。
2. **房间完成：** 将种子带到出口花或种植点。
3. **中心花园进展：** 种子让中心花园恢复一处颜色、声音、路径或植物。
4. **区域进展：** 一组种子恢复区域入口或新的主题空间。
5. **整体进展：** 中心花园从沉睡逐步变成可停留、可回看、可选择章节的空间。

### Difficulty Curve

难度曲线以“规则理解”递增，不以操作难度递增。

| Phase | Complexity | Design Target |
| --- | --- | --- |
| First 10 Minutes | 单规则，强反馈 | 玩家理解世界切换会改变路径 |
| Early Game | 单规则变体 | 玩家能预测藤蔓、花门、出口花状态 |
| Mid Game | 双规则组合 | 玩家开始先观察再行动 |
| Late Game | 区域规则组合 | 玩家能读懂多个对象的关系 |
| Optional | 高密度组合 | 给核心解谜玩家更多满足，不阻挡主线 |

平衡指标：

- 教学房间完成率目标：80% 以上试玩玩家在 5 分钟内完成。
- 第一次世界切换理解率目标：70% 以上玩家能用自己的话说明“切换改变路径或对象状态”。
- 第一颗种子回报感目标：70% 以上玩家能表达类似“我修复了这里”的感受。
- 因看不懂关键对象而卡住的玩家比例目标：低于 20%。

### Economy and Resources

主线不设置传统经济。核心资源只有：

- **发光种子：** 主线进度与修复反馈标记，不消费，不交易。
- **花园恢复度：** 中心花园视觉和空间变化，不作为数值压力展示。
- **可选纪念物：** 花园明信片、种子图鉴或环境片段，只服务收藏和回看。

不使用金币、能量、体力、材料合成、商店升级或数值成长，避免把轻量修复体验变成管理游戏。

## Level Design Framework

### Level Types

| Level Type | Purpose | Content Rules |
| --- | --- | --- |
| Center Garden | 章节入口、进度展示、情绪回报 | 每次种植产生永久变化；不做复杂经营 |
| Tutorial Room | 教一个动作或一个对象规则 | 2-4 分钟；目标可见；提示强 |
| Motif Room | 探索一个区域主规则 | 4-8 分钟；一个新规则或一个变体 |
| Combination Room | 组合已学规则 | 6-10 分钟；最多两个主规则 |
| Restoration Room | 区域收束与高回报 | 低难度，强调恢复仪式 |
| Optional Room | 给深度玩家额外挑战或收藏 | 不阻挡主线；可跳过 |

### Level Progression

第一垂直切片房间结构：

1. 入口区：玩家进入小花园，看到远处发光种子。
2. 世界切换教学：玩家切换到 Bloom，藤蔓桥展开。
3. 路线理解：玩家过桥接近花门。
4. 花门反馈：花门在正确状态或携带种子后醒来。
5. 种子收集：种子跟随玩家，出口花被点亮。
6. 返回出口：玩家带种子到出口花。
7. 种植仪式：种子落地，开出第一朵恢复花。

标准房间设计检查：

- 入口是否让目标或方向可读？
- 当前房间的“小魔法”能否一句话说明？
- 玩家第一次切换后是否看到有意义的变化？
- 收集种子后是否知道去哪里？
- 完成后是否有与房间主题相关的恢复反馈？

## Art and Audio Direction

### Art Style

美术方向是 2D 俯视角、轻绘本感、柔和手绘风。枯萎世界低饱和、安静、偏冷或灰绿，但不阴森；盛放世界温暖、明亮，带花瓣、微光、柔和运动和更清晰的生命感。

角色和种子应可爱但不低幼。小精灵需要显得迷路但勇敢；发光种子需要像短暂同行的小伙伴，而不是普通可拾取道具。

关键视觉规则：

- 关键对象状态至少用两种维度表达：颜色 + 形状、颜色 + 动画、轮廓 + 发光、动作 + 音频。
- 世界切换必须有从玩家或房间中心扩散的盛放涟漪。
- 藤蔓桥、花门、出口花、种子和种植点优先拥有独立动画。
- 画面密度服务可读性；装饰植物不能掩盖路线和交互对象。

[ASSUMPTION A4: AI 可用于概念探索、占位素材和风格试验，但正式资产需要统一调色、轮廓、动画节奏和导入规范，避免拼贴感。]

### Audio and Music

音频应让玩家听见花园状态变化。

| Audio Element | Withered | Bloom | Gameplay Function |
| --- | --- | --- | --- |
| Ambient Bed | 轻风、稀疏木质声、低密度铃声 | 柔和旋律、风铃、鸟鸣、植物细响 | 区分世界状态 |
| World Switch | 低到高的微光扫过声 | 高到低的柔和回落声 | 确认切换 |
| Vine Bridge | 干枯摩擦或叶片轻响 | 藤蔓舒展声 | 确认通路变化 |
| Flower Door | 沉睡呼吸声 | 花瓣打开声 | 确认阻挡解除 |
| Seed Collect | 小铃声、光点回应 | 微弱跟随音 | 强化情感对象 |
| Planting | 2-4 秒恢复音、音乐层增强 | 花园声场变丰满 | 完成和奖励 |

不需要正式配音。小精灵可以使用少量非语言反应音，避免文本和对白抢走解谜节奏。

## Technical Specifications

### Performance Requirements

| Target | Requirement | Measurement |
| --- | --- | --- |
| PC / Steam | 1080p 下稳定 60 FPS | 10 分钟核心循环试玩，无明显掉帧 |
| Web Demo | 主流桌面浏览器稳定 60 FPS，弱设备可接受 30 FPS 下限 | 首房间完整流程 |
| Load | Web demo 首次可交互目标 30 秒内 | 普通宽带环境人工测试 |
| Switch Feel | 世界切换无加载屏，转场 0.45-0.75 秒 | 玩家可感知连续性 |
| Input | 移动和切换输入响应目标低于 100 ms | 手感测试 |

### Platform-Specific Details

PC / Steam 是首发目标，必须支持键盘和手柄。Web demo 用于试玩、分享和早期反馈，内容应控制在一个完整垂直切片或少量房间，避免加载体积过大。

移动端是后续验证方向，不进入垂直切片承诺。若未来支持移动端，需要重新验证触控移动、按钮布局、屏幕可读性、短时会话和商店发行策略。

目标引擎候选为 Godot 2D。GDD 只规定玩家体验和平台目标；具体对象架构、状态同步、资源导入、存档和构建管线应在后续架构文档中定义。

### Asset Requirements

垂直切片最低资产清单：

- 小精灵：待机、移动、互动、种植反应。
- 房间地块：枯萎 / 盛放两套状态。
- 藤蔓桥：卷起、展开、可通行状态。
- 花门：睡眠、唤醒、开启状态。
- 发光种子：待收集、收集、跟随、种下。
- 出口花：关闭、可用、打开。
- 中心花园小块：未恢复、恢复中、恢复完成。
- 世界切换涟漪、花瓣路标、微光轨迹、种植粒子。
- 音频：双世界环境层、切换、桥、门、种子、种植完成。

## Success Metrics

### Technical Metrics

- PC 垂直切片 1080p 稳定 60 FPS。
- Web demo 首次可交互目标 30 秒内。
- 世界切换转场 0.45-0.75 秒，无加载屏和明显卡顿。
- 10 分钟试玩内不出现阻断核心循环的错误。
- 键盘和手柄均可完成完整房间闭环。

### Gameplay Metrics

- 80% 以上目标玩家能在 5 分钟内完成第一房间。
- 70% 以上玩家能在试玩后描述“切换世界会改变路径或对象状态”。
- 70% 以上玩家能在种下第一颗种子后表达明确的修复回报感。
- 少于 20% 的玩家因看不懂藤蔓桥、花门、出口花或种子用途而卡住。
- 玩家首次主动使用世界切换后，能在 10 秒内观察到至少一个有意义变化。
- 试玩反馈能明确指向下一阶段优先级：视觉可读性、谜题结构、音频反馈、移动手感或恢复回报。

## Assumptions and Dependencies

### Assumptions Index

- **A1:** 完整版 v1 以 4 个区域、16-24 个主线房间、4 个区域恢复节点为规划上限；该数字用于设计和生产估算，不是当前垂直切片承诺。
- **A2:** 第一版以键盘和手柄为主，触控适配在核心循环验证后再评估。
- **A3:** 完整版可允许玩家在连续使用多层提示后跳过非教程主线谜题，但垂直切片先不实现跳过，以免掩盖核心规则可读性问题。
- **A4:** AI 可用于概念探索、占位素材和风格试验，但正式资产需要统一调色、轮廓、动画节奏和导入规范，避免拼贴感。

### Dependencies

- Godot 2D 或等价 2D 引擎工作流的最终确认。
- 一套统一视觉风格板：枯萎、盛放、可交互、已完成、不可通行。
- 最小音频方向测试：枯萎 / 盛放环境层、世界切换、种植完成。
- 至少 5 名目标玩家完成垂直切片试玩，用于验证理解度和回报感。
- 后续架构文档定义世界状态、对象响应、资源导入、存档和构建管线。
